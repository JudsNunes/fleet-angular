## 0. Princípios do Open Spec Development (Passo 2 Refatorado)

O contrato da API é a **única fonte de verdade** antes da escrita de qualquer código.

1. **Design First:** `openapi.yaml` é escrito e aprovado antes de Controllers (backend) ou Services (Angular).
2. **Code-Generation:**
   - Frontend: `@openapitools/openapi-generator-cli` gera modelos TypeScript + serviços Angular (HttpClient).
   - Backend: Validação estrita de interfaces geradas contra schemas definidos.
3. **Evolução Segura:** Qualquer quebra de contrato falha no build de CI/CD.

### Módulos e Contratos OpenAPI

| Módulo | Schemas | Endpoints-chave | Notas |
|--------|---------|-----------------|-------|
| **Identidade/Multi-Tenancy** | `Tenant`, `User`, `Role` | `POST /api/v1/auth/token`, `GET /api/v1/admin/tenants` | `bearerAuth` (JWT), header `X-Tenant-ID` |
| **Gestão de Ativos** | `Vehicle`, `Driver`, `VehicleStatusEnum` | `GET /api/v1/vehicles`, `POST /api/v1/vehicles` | Regex patterns (placa Mercosul), `x-spring-data-pageable` |
| **Telemetria/Rota** | `GeoLocationEvent` | AsyncAPI: `topic/telemetry/{tenantId}/{vehicleId}` | WebSocket (STOMP), timestamp ISO-8601 |
| **Manutenção/Custos** | `MaintenanceOrder`, `Expense`, `ProblemDetail` | `POST /api/v1/maintenance/orders`, `GET /api/v1/analytics/cpk` | RFC 7807 para erros de regra de negócio |

### YAML Base (Referência)

```yaml
openapi: 3.1.0
info:
  title: Fleet SaaS API
  version: 1.0.0-MVP
paths:
  /api/v1/vehicles:
    post:
      summary: Cadastra um novo veículo na frota
      security:
        - bearerAuth: []
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/VehicleRequest'
      responses:
        '201':
          description: Veículo criado com sucesso
        '400':
          description: Erro de validação de negócio (RFC 7807)
          content:
            application/problem+json:
              schema:
                $ref: '#/components/schemas/ProblemDetail'
components:
  schemas:
    VehicleRequest:
      type: object
      required: [licensePlate, chassis, capacity]
      properties:
        licensePlate:
          type: string
          pattern: '^[A-Z]{3}[0-9][A-Z0-9][0-9]{2}$'
        chassis:
          type: string
          minLength: 17
          maxLength: 17
```

**Cenários BDD:**

```gherkin
# language: pt
Cenário: Arquitetura de standalone components
  Dado que o projeto usa Angular 22
  E todos os componentes são standalone (sem NgModules)
  Quando um novo componente é criado
  Então ele declara seus próprios imports no decorator @Component
  E não existe nenhum NgModule no projeto

Cenário: State management com Signals e RxJS
  Dado que a aplicação precisa de estado síncrono (UI) e assíncrono (HTTP)
  Quando o estado é de UI (loading, sidebar, filtros)
  Então usa-se Angular Signals
  Quando o estado é de dados assíncronos (HTTP, WebSocket)
  Então usa-se RxJS BehaviorSubject + Observable

Cenário: Lazy loading de features
  Dado que existem 8 módulos de feature (vehicles, drivers, trips, etc.)
  Quando o usuário navega para uma feature
  Então o código da feature é carregado sob demanda (lazy)
  E o bundle inicial contém apenas core + layout + dashboard

Cenário: Interceptadores HTTP
  Dado que a API retorna erros com fieldErrors para validação
  Quando ocorre um erro 400 com fieldErrors
  Então o GlobalErrorInterceptor delega para FormValidationService
  E o FormValidationService aplica os erros nos controles do Reactive Form
```

- Projetar estrutura de pastas (feature-based)

```
    └── audit-logs/      # Read-only com filtros
```

### 3.2 Decisões Chave

| Decisão | Escolha | Rationale |
|---------|---------|-----------|
| Components | Standalone (sem NgModules) | Angular 22 default |
| State (UI) | Angular Signals | Loading, filtros, sidebar |
| State (HTTP) | RxJS BehaviorSubject | Paginação + debounce + cancellation |
| UI Library | Angular Material | MatTable, MatPaginator, MatSnackBar |
| Forms | Reactive Forms | Bind de fieldErrors via setErrors() |
| Routing | Lazy-loaded features | `loadComponent` por feature |
| Config | Injection token ENVIRONMENT | apiUrl não hardcoded |

### 3.3 Service Layer

```typescript
// ApiBaseService — abstrato genérico
export abstract class ApiBaseService<TResponse, TRequest> {
  protected abstract readonly basePath: string;
  getAll(params?): Observable<TResponse[]>
  getPage(page, size, params?): Observable<PageableResponse<TResponse>>
  getById(id): Observable<TResponse>
  create(body): Observable<TResponse>
  update(id, body): Observable<TResponse>
  delete(id): Observable<void>
}
```

### 3.4 Error Handling

```
HTTP 400 + fieldErrors → errorInterceptor → FormValidationService → setErrors()
HTTP 404 → errorInterceptor → NotificationService → toast "Não encontrado"
HTTP 500 → errorInterceptor → NotificationService → toast "Erro interno"
```
